# Audit Plan - Sistem Akademik

## Objective

Review the academic system implementation against the business requirements in `kelompok10_akademik_b (1).md` and the 10 requested quality gates:

1. Code quality and structure
2. Framework and library best practices
3. UI/UX compliance
4. Performance optimization
5. Security
6. API integration
7. File and asset management
8. Documentation and comments
9. Testing coverage
10. Deployment readiness

## Scope Assumption

Default assumption: audit the actual SvelteKit project in this repository and use `kelompok10_akademik_b (1).md` as the functional baseline.

If the user instead wants a markdown-only review, the audit changes substantially: the output would focus on requirement completeness, clarity, and implementation readiness of the case-study document rather than code quality, security, API behavior, UI, and deployment.

## Planned Review Phases

### Phase 1 - Requirement Mapping

- Map the case-study narrative to implemented entities and workflows:
  - ruang kelas
  - jadwal penggunaan ruang
  - enrollment
  - KRS
  - nilai and huruf mutu
  - IPK expectations
- Identify mismatches between the document and schema/API behavior.

### Phase 2 - Architecture and Code Audit

- Review page structure, reusable components, service layer, server utilities, and route handlers.
- Check naming consistency, duplication, typing quality, abstraction boundaries, and dead or misleading UI features.

### Phase 3 - UX, Security, and Reliability Audit

- Check client loading patterns, navigation behavior, accessibility basics, error handling, auth model, validation depth, and API consistency.
- Highlight production-impacting gaps first.

### Phase 4 - Test and Deploy Audit

- Verify current test surface versus application complexity.
- Review environment setup, adapter choice, migration strategy, CI/CD readiness, and production documentation.

### Phase 5 - Deliverable Format

- Produce findings grouped by the 10 requested categories.
- For each finding include:
  - Severity
  - Location
  - Issue
  - Recommendation
- End with:
  - overall assessment
  - readiness score out of 10
  - prioritized action list

## Likely High-Priority Themes Already Identified

- Client-side `onMount` data loading across many pages instead of SvelteKit `load`
- Shared Basic Auth without user roles or route-level authorization
- Search and sorting affordances not fully wired end-to-end in some screens
- KRS workflow not fully aligned with enrollment scheduling model in the case study
- Stored `Mahasiswa.ipk` field not clearly synchronized with grade changes
- Thin automated test coverage for the size of the API/UI surface
- `adapter-auto` and missing migrations/CI reduce deployment readiness
